# Unitree G1 開発環境構築 & 接続手順

Mac / Windows 共通の開発環境セットアップと、G1 への接続手順をまとめたガイドです。

---

## 1. 開発環境の構築

### 1.1 必要なもの

| 項目 | 要件 |
|---|---|
| OS | macOS / Windows 11（WSL2）/ Ubuntu 22.04 |
| Python | **3.10**（必須。3.11 以降は不可） |
| LAN ケーブル | G1 との直結用 |
| ストレージ | 5GB 以上の空き |

> **なぜ Python 3.10 なのか？**
> G1 の通信に使う `cyclonedds==0.10.2` のビルド済みバイナリが Python 3.7〜3.10 までしか提供されていないためです。3.11 以降ではソースビルドが必要になり、トラブルの原因になります。
> なお、G1 内部は Python 3.8 ですが、開発 PC 側と合わせる必要はありません（DDS ネットワークプロトコルでの通信のため）。

### 1.2 Python 3.10 のインストール

#### macOS

```bash
brew install python@3.10
echo 'export PATH="/opt/homebrew/opt/python@3.10/bin:$PATH"' >> ~/.zshrc
source ~/.zshrc
python3.10 --version   # Python 3.10.x と表示されれば OK
```

#### Windows（WSL2）

WSL2 の Ubuntu 22.04 にはデフォルトで Python 3.10 が入っています。

```bash
# WSL2 のインストール（PowerShell を管理者として実行）
wsl --install -d Ubuntu-22.04

# Ubuntu 起動後に確認
python3 --version   # Python 3.10.x と表示されれば OK
```

### 1.3 リポジトリの取得

```bash
git clone <リポジトリURL>
cd unitree_g1_hackathon
```

### 1.4 仮想環境の作成と SDK インストール

#### macOS

```bash
python3.10 -m venv .venv
source .venv/bin/activate
pip install -e ./unitree_sdk2_python
```

#### Windows（WSL2）

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -e ./unitree_sdk2_python
```

> `unitree_sdk2_python/` に含まれる `setup.py` が `cyclonedds==0.10.2`, `numpy`, `opencv-python` を自動でインストールします。

### 1.5 インストール確認

```bash
python -c "from unitree_sdk2py.core.channel import ChannelFactoryInitialize; print('SDK OK!')"
```

`SDK OK!` と表示されれば完了です。

## 2. G1 との接続

### 2.1 ネットワーク構成

開発 PC と G1 は Ethernet（有線 LAN）で直結します。

| デバイス | IP アドレス | 役割 |
|---|---|---|
| 開発 PC（あなた） | 192.168.123.99 | 外部開発環境 |
| Jetson Orin（G1 内部） | 192.168.123.164 | SSH 先 |
| RockChip | 192.168.123.161 | ロコモーション制御 |
| LiDAR | 192.168.123.120 | 点群データ |

> **SSH と SDK（DDS）通信は別物です**
>
> | | SSH | SDK（DDS 通信） |
> |---|---|---|
> | 接続先 | Jetson Orin (`192.168.123.164`) | RockChip (`192.168.123.161`) |
> | プロトコル | TCP (port 22) | UDP（DDS マルチキャスト） |
> | 仕組み | 通常の IP 通信 | CycloneDDS がインターフェースを指定して通信 |
>
> **SSH が通っても SDK の通信が通るとは限りません。**
> SDK は DDS プロトコル（UDP マルチキャスト）で RockChip と直接通信するため、インターフェース名の指定やネットワーク設定が正しくないと動きません。

### 2.2 IP アドレスの設定

LAN ケーブルを接続したら、開発 PC に IP アドレスを設定します。

#### macOS

まずインターフェース名を確認します：

```bash
ifconfig | grep -B2 "1000baseT"
```

`en11` のようなインターフェース名が表示されます（環境によって異なる）。

```bash
sudo ifconfig en11 192.168.123.99 netmask 255.255.255.0
```

> **注意:** macOS では LAN ケーブルの抜き差しやスリープ復帰で IP 設定がリセットされることがあります。`ping` が通らなくなったら再設定してください。

#### Windows（WSL2）

Windows 側の**物理イーサネットアダプタ**に IP を設定します。

> **注意:** `vEthernet (WSL)` は WSL2 内部通信用の仮想 NIC です。ここに設定しても G1 に届きません。必ず物理 LAN ポート（「イーサネット」「イーサネット 3」等）に設定してください。

**方法 A: GUI で設定**

1. 設定 → ネットワークとインターネット → イーサネット → IP 設定 → 編集
2. 手動に切り替え、IPv4 を ON
3. 以下を入力：

| 項目 | 値 |
|---|---|
| IP アドレス | `192.168.123.99` |
| サブネットマスク | `255.255.255.0` |
| デフォルトゲートウェイ | （空欄） |

**方法 B: PowerShell（管理者）で設定**

```powershell
# インターフェース名を確認（物理NICを探す）
Get-NetAdapter | Where-Object {$_.Status -eq "Up"}

# IP を設定（"イーサネット" は物理NICの名前に合わせて変更）
New-NetIPAddress -InterfaceAlias "イーサネット" -IPAddress 192.168.123.99 -PrefixLength 24
```

### 2.3 疎通確認

```bash
ping -c 3 192.168.123.164    # macOS / WSL2
```

```
64 bytes from 192.168.123.164: icmp_seq=0 ttl=64 time=0.437 ms
64 bytes from 192.168.123.164: icmp_seq=1 ttl=64 time=0.623 ms
64 bytes from 192.168.123.164: icmp_seq=2 ttl=64 time=0.529 ms
```

3 パケットとも応答があれば OK です。

### 2.4 SDK 接続テスト

```bash
source .venv/bin/activate
python test_connection.py <インターフェース名>
```

- macOS: `python test_connection.py en11`（インターフェース名は環境依存）
- WSL2: `python test_connection.py eth0`（`ip a` で確認）

```
Connecting to G1 via interface: en11
Waiting for LowState data...
=== G1 LowState received ===
IMU accelerometer: x=-0.210, y=-0.160, z=9.710
Motor count (non-zero q): 26
Connection OK!
```

IMU と関節データが表示されれば、G1 との DDS 通信が成功しています。

## 4. トラブルシューティング

| 症状 | 原因 | 対処 |
|---|---|---|
| `ping` が通らない | IP 未設定 or LAN ケーブル未接続 | Step 2.2 をやり直す |
| `SDK OK!` が出ない | 仮想環境未有効化 | `source .venv/bin/activate` を実行 |
| `No data received` | インターフェース名が違う | macOS: `ifconfig` / WSL2: `ip a` で確認 |
| `send request error` | DDS 通信が届いていない | 下記「SSH は通るが SDK が動かない場合」を参照 |
| `does not match an available interface` | NIC 名が変わった（WSL2 再起動後に多い） | `ip addr show` で `192.168.123.99` の NIC 名を確認し直す |
| macOS で突然通信が切れた | スリープやケーブル抜き差しで IP がリセット | `sudo ifconfig en11 192.168.123.99 netmask 255.255.255.0` を再実行 |
| `cyclonedds` インストールエラー | Python 3.11 以降を使っている | Python 3.10 で仮想環境を作り直す |
| WSL2 から `ping` が通らない | WSL2 のネットワーク設定 | `.wslconfig` に `networkingMode=mirrored` を追加して WSL 再起動 |
| 依存関係エラー（apt） | パッケージの不整合 | `sudo apt --fix-broken install -y` を実行してから再セットアップ |

### SSH は通るが SDK（DDS）が動かない場合

SSH（TCP）と SDK（DDS/UDP マルチキャスト）は別の通信経路です。SSH が通っても SDK が動かないケースがあります。

**チェックリスト：**

1. **インターフェース名は正しいか？**
   ```bash
   # macOS
   ifconfig | grep -B2 "1000baseT"
   # WSL2
   ip a | grep 192.168.123
   ```
   SDK に渡すインターフェース名（`en11`, `eth0` 等）が、実際に `192.168.123.99` が設定されているものと一致しているか確認。

2. **IP アドレスがリセットされていないか？**（macOS）
   ```bash
   ifconfig en11 | grep inet
   ```
   `192.168.123.99` が表示されなければ再設定が必要。

3. **WSL2 のミラーモードは有効か？**（Windows）
   WSL2 はデフォルトで NAT モードのため、UDP マルチキャストが物理 LAN に届きません。ミラーモードを有効にする必要があります（下記参照）。

4. **G1 側の制御プロセスは起動しているか？**
   SSH で G1 に入って確認：
   ```bash
   ssh unitree@192.168.123.164
   # パスワード: 123
   ps aux | grep -E "sport|loco|dds" | grep -v grep
   ```
   何も表示されなければ G1 側のプロセスが起動していない可能性があります。G1 の再起動を検討してください。

5. **`test_connection.py` でデータが取れるか？**
   ```bash
   python test_connection.py <インターフェース名>
   ```
   `Connection OK!` が出れば DDS 通信は正常です。`No data received` なら上記 1〜4 を確認してください。

### WSL2 ネットワーク設定（Windows のみ）

WSL2 から G1 と DDS 通信するには、ミラーモードの設定が**必須**です。

> **なぜ必要か？**
> WSL2 はデフォルトで NAT（ネットワーク変換）を介して通信します。SSH などの TCP 通信は NAT を通過できますが、DDS が使う UDP マルチキャストは NAT を越えられません。ミラーモードにすると、WSL2 が Windows と同じネットワークインターフェースを共有するため、DDS 通信が物理 LAN に届くようになります。

**設定手順：**

1. PowerShell で `.wslconfig` を作成（既にあれば手動で編集）：

```powershell
@"
[wsl2]
networkingMode=mirrored
"@ | Set-Content $env:USERPROFILE\.wslconfig
```

確認：

```powershell
cat $env:USERPROFILE\.wslconfig
```

2. PowerShell で WSL を再起動：

```powershell
wsl --shutdown
```

3. Ubuntu を起動し直して、インターフェースを確認：

```bash
ip a | grep 192.168.123
```

Windows 側で設定した `192.168.123.99` が見えていれば OK です。

> **注意:** WSL2 再起動後に NIC 名が変わることがあります（`eth0` → `eth1` 等）。SDK 実行時にインターフェース名のエラーが出たら `ip addr show` で再確認してください。
